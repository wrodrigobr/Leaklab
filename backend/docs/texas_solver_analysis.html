1. Arquitetura da lib (o que já é bom)

ComponenteComo é
AlgoritmoDiscounted CFR (solver.rs): α por schedule √t·t, β=0.5 (downweight de regret negativo), γ decaimento cúbico com reset em potências de 4. Regret-matching com clipping. Exploitability = média dos EVs da melhor resposta (MES), recalc a cada 10 iters.
ParalelismoRayon, fork-join por ação (cada subárvore de ação vira task). MutexLike = UnsafeCell (sem lock real; sincronização manual por particionar escrita em rows distintas) → baixo overhead, mas exige a invariante da árvore.
SIMDSIMD explícito em sliceop.rs/utility.rs (unroll 8-wide, FMA) + path WASM SIMD128. Hot loops escalares contam com autovetorização. Sem intrínsecos AVX2/AVX-512 explícitos no x86.
MemóriaCompressão opcional 16-bit (i16 regrets / u16 strategy) — metade da RAM; alocador stack custom (buffers de 1MB, LIFO). Cap prático no wrapper: 6GB (≈60bb).
EvalForça de mão algorítmica (rankset/flush/straight por bitops) + binary-search numa tabela esparsa de 4824 entradas; precomputa a força por board. Suit-agnostic.
BunchingCorreção de cartas mortas dos foldados preflop (até 4 em 6-max). Existe e é testado contra o GW, mas NÃO é usado pelo wrapper. Custo alto (3-fold já pesa; 4-fold pede GBs).
Featuresrayon ✓, bincode ✓, zstd ✗ (off), custom-alloc ✗ (nightly). Vocês usam só rayon.

2. P0 — Bugs de correção no wrapper (prioridade máxima)

2.1 Estratégia do jogador ERRADO quando o hero é IP

O solver_cli/main.rs (linha 201-207) sempre lê o player 0 = OOP: game.strategy(), private_cards(0), expected_values(0). E o gto_solver.py monta oop_range = vs_position, ip_range = position(hero). Resultado: quando o hero está IP (ex.: BTN dando c-bet vs BB), o nó armazenado é a estratégia do vilão (OOP) — não do hero. No teste de validação, o "check 81% do BTN" era, na verdade, a linha do BB.

Confirmado independentemente na lib: a estratégia do IP não é lida direto no root — precisa navegar (jogar a carta/ação) até o nó do IP, ou derivar via cfvalues_ip. Por isso o wrapper "facilmente" só pega o OOP.

Fix: no gto_solver.py, determinar quem é OOP/IP de fato pela posição postflop (SB/BB são OOP vs late; o último a agir é IP), atribuir as ranges corretas, e retornar a estratégia do HERO. No main.rs, aceitar um flag hero_is_ip (ou hero_player ∈ {0,1}) e, se IP, navegar pro nó do IP (após o check do OOP) e ler strategy()/private_cards(1) lá. esforço M risco médio crítico

2.2 facingSize em CHIPS tratado como bb

O pipeline entrega facingSize em fichas (ex.: 240 com bb=120 → 2.0bb), mas o solver navega o facing em bb (main.rs: facing_chips = facing_size_bb*100). Passar fichas como bb leva ao nó errado (ou a nenhum). Hoje só facing=0 é seguro. (O mesmo valor cru também degenera o bet_bucket do hash — mas como store e read usam o mesmo valor, o match não quebra; só a navegação do solve quebra.)

Fix: converter facing_size_bb = facingSize / bb ao chamar o lookup_gto, e padronizar o hash pra usar facing em bb (cuidado: invalida nós GW antigos com facing cru — migrar). esforço S risco baixo

2.3 Ranges genéricas em vez das reais do GW

O solve usa _DEFAULT_RANGES (ranges largas fixas por posição) pro OOP/IP. Mas nós temos as ranges reais capturadas do GW (leaklab_gto_ranges.json) — e provamos no comparativo GW×Texas que, com elas, o Texas dá defesa GTO-coerente (fold 47-55% / call / raise por textura). Com ranges genéricas a precisão cai.

Fix: derivar oop_range/ip_range das ranges preflop capturadas (RFI do agressor + range de defesa/3bet do caller, pelo bucket de stack), como já fiz no script de comparação. esforço S–M risco baixo grande ganho de qualidade

Implicação imediata: a reativação do Texas postflop (commit e5dc299) funciona mecanicamente e as travas (depth/SPR) evitam recs absurdas, mas até 2.1–2.3 serem corrigidos ela pode rotular contra a estratégia do jogador errado / range errada em parte dos spots. Não rodar o batch de NULLs nem confiar nos labels postflop do Texas antes desses fixes.

6. Licença e manutenção

  • AGPL-3.0 + arquivada (out/2023): sem upstream, sem patches. Vocês pinam o commit 9d1509f (estável).
  • Copyleft de rede: como produto hosted, a AGPL obriga oferecer o código-fonte (incl. modificações) a quem usa via rede — ponto jurídico a resolver com o jurídico antes de investir pesado/depender mais.
  • Recomendação: forkar pra org de vocês (protege contra remoção do repo + permite os patches P1/P2), documentar a AGPL, e só então atacar as melhorias da lib. As correções P0 são todas no nosso wrapper (não exigem fork).

Próximo passo sugerido

Atacar os 3 fixes P0 (correto antes de rápido): eles destravam o batch dos 178 postflop NULLs com dados confiáveis. Os 3 são no nosso código (sem fork, sem mexer na lib). Depois, P1 (2 sizes + bunching + cache) eleva a qualidade; P2 (AVX2 → Predictive CFR+) é o ganho de performance com melhor risco.

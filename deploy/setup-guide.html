Guia de Deploy detalhado — GrindLab

Passo a passo, com um teste em cada etapa. Cloudflare (frontend + proxy) · Neon (Postgres) · Hetzner (backend). Domínios grindlabpoker.com.br (canônico) e grindlabpoker.com (→ 301).

Visão geral

Usuário (navegador) │ ▼ ┌──────────────── Cloudflare ────────────────┐ │ Pages (hospeda o frontend) + Proxy/CDN │ └──────┬───────────────────────────┬─────────┘ │ grindlabpoker.com.br │ api.grindlabpoker.com.br ▼ ▼ [ App React ] ┌──────────────────┐ TLS ┌──────────────┐ (borda CF) │ Hetzner #1 CX23 │ ─────▶ │ Neon │ │ Nginx → Gunicorn │ │ PostgreSQL │ │ (Flask, Docker) │ │ (backup auto)│ └────────┬─────────┘ └──────────────┘ │ rede privada Hetzner (10.0.0.x) ▼ ┌──────────────────┐ │ Hetzner #2 CX43+ │ solver GTO (AGPL), │ (solver, privado)│ nunca público └──────────────────┘
4 painéis + 1 terminal. Você vai usar Cloudflare, Neon, Hetzner e Stripe no navegador, e um terminal SSH na VPS. Faça na ordem 0→8 e só avance quando o "🧪 Teste" da etapa passar.

Como ler este guia

  • ASSIM = valor que VOCÊ substitui (IP, senha, chave). Nunca cole literal.
  • Blocos pretos = comandos pra colar no terminal (linhas cinza são comentários).
  • 🧪 Teste = como verificar que a etapa funcionou antes de seguir.
  • ⛔ = passo onde dá pra se trancar fora / quebrar — leia 2×.

0.2 Preparar o PC (Windows) — chave SSH

  1. Abra o PowerShell (Iniciar → "PowerShell").
  2. Gere a chave: 
    ssh-keygen -t ed25519 -C "grindlab-deploy"
# ENTER em todas as perguntas
  3. Mostre a chave pública (vai colá-la no Hetzner): 
    type $env:USERPROFILE\.ssh\id_ed25519.pub
🧪 Teste: o comando imprime uma linha começando com ssh-ed25519 AAAA… terminando em grindlab-deploy. Se imprimiu, sua chave está pronta.

1Cloudflare — domínios ~15 min + propagação

Colocar os 2 domínios na Cloudflare. O .com já está pronto ✔; falta o .com.br.

1.1 Adicionar a zona .com.br

  1. dash.cloudflare.com → + Add a domain → grindlabpoker.com.br → Continue.
  2. Plano Free → Continue → (revisa DNS) → Continue.
  3. Anote os 2 nameservers que aparecem (ex.: aria.ns.cloudflare.com).

1.2 Trocar os nameservers no registro.br

  1. registro.br → login → domínio grindlabpoker.com.br → DNS → Alterar servidores DNS.
  2. Cole os 2 nameservers da Cloudflare → salve. Volte à Cloudflare → Check nameservers.
🧪 Teste (no PowerShell): 
nslookup -type=NS grindlabpoker.com.br
Quando aparecer os nameservers *.ns.cloudflare.com, propagou. (Também dá pra conferir em dnschecker.org → tipo NS.) Pode levar de minutos a horas — siga as outras fases enquanto isso.

1.3 Adicionar a zona .com + redirect 301

  1. Repita 1.1 para grindlabpoker.com (deve ficar Active rápido, já que você já trocou os NS dele).
  2. Selecione a zona grindlabpoker.com → Rules → Redirect Rules → Create rule.
  3. When: Hostname contains grindlabpoker.com. Then: Type Dynamic, Expression: 
    concat("https://grindlabpoker.com.br", http.request.uri.path)
  4. Status 301, Preserve query string ON → Deploy.
🧪 Teste (depois de ambas Active): 
curl.exe -I https://grindlabpoker.com
Deve responder HTTP/.. 301 com location: https://grindlabpoker.com.br/.

2Neon — banco PostgreSQL ~8 min

O banco. Backup automático incluso.

2.1 Criar o projeto

  1. console.neon.tech → Create project.
  2. Name: grindlab. Postgres: última versão.
  3. Region: a mais perto da Hetzner que você vai escolher na Fase 3 — se for Hetzner Ashburn (US-East), use Neon AWS us-east-1; se for Hetzner Alemanha, use Neon AWS eu-central-1. (Banco e backend devem ficar na mesma região.)
  4. Database name: grindlab → Create.

2.2 Copiar a connection string

  1. No projeto → Connect → opção Pooled connection.
  2. Copie a string (formato): 
    postgresql://usuario:senha@ep-xxxx-pooler.regiao.aws.neon.tech/grindlab?sslmode=require
🧪 Teste: no Neon, abra o SQL Editor e rode SELECT now(); → retorna a data/hora atual. (A conexão real do app é testada na Fase 4.)

3Hetzner — o servidor (VPS) ~25 min

3.1 Criar o servidor

  1. console.hetzner.cloud → seu projeto → Add Server.
  2. Location: a mais perto do Brasil que tiver a opção desejada — Ashburn (US-East) tem boa latência; Alemanha/Finlândia também servem (o frontend já está na borda da Cloudflare, então o impacto é pequeno).
  3. Image: Ubuntu 24.04.
  4. Type: aba Shared vCPU → linha CX (Intel) (mais barata que AMD/CPX). Esta é a máquina só do backend (o solver fica na 2ª máquina, Fase 9), então pode ser enxuta: CX23 ⭐ (2 vCPU / 4 GB / 40 GB, ~US$6,49/mês). Obs.: a análise de hand history usa CPU; se muitos uploads simultâneos ficarem lentos, suba pra CX33 (4/8 GB) com 1 clique.
  5. SSH key: Add SSH key → cole a chave pública (Fase 0.2) → dê um nome.
  6. Name: grindlab-api → Create & Buy now.
  7. Anote o IP público (ex.: 5.6.7.8).
🧪 Teste (PowerShell): 
ssh root@5.6.7.8
Você entra sem pedir senha (usou a chave) e vê root@grindlab-api:~#. Se pedir senha ou recusar, a chave não foi associada — recrie o servidor com a chave certa.
Arquitetura de 2 máquinas (sua escolha): esta caixa (CX33) roda só o backend. O solver fica numa 2ª máquina maior (RAM p/ os solves) — assim um solve nunca deixa o site lento. Veja a Fase 9 abaixo para a 2ª máquina. As duas ficam na mesma rede privada do Hetzner e conversam por IP interno; o solver nunca fica público.

3.2 Firewall do Hetzner (1ª camada — só Cloudflare + seu SSH)

No painel Hetzner, Firewalls → Create Firewall:

  1. Inbound rule 1 — SSH: TCP, porta 22, Source = SEU_IP (veja em ifconfig.me) com /32.
  2. Inbound rule 2 — HTTPS: TCP, porta 443, Source = os ranges da Cloudflare (cole a lista de cloudflare.com/ips-v4 e /ips-v6).
  3. Inbound rule 3 — HTTP: TCP, porta 80, mesmos ranges Cloudflare.
  4. Apply to: o servidor grindlab-api → Create Firewall.
Colar ~15 ranges na mão é chato — se preferir, pule para um firewall só de SSH aqui e deixe o ufw (3.3) restringir 80/443 à Cloudflare via script. Uma das duas camadas já basta; ter as duas é o ideal.

3.3 Endurecer o sistema (como root)

  1. Usuário dedicado + cópia da sua chave: 
    adduser deploy && usermod -aG sudo deploy
rsync --archive --chown=deploy:deploy ~/.ssh /home/deploy
  2. Patches + ferramentas: 
    apt update && apt -y upgrade
apt -y install ufw fail2ban unattended-upgrades
dpkg-reconfigure -plow unattended-upgrades   # escolha <Yes>
  3. Firewall do SO (2ª camada): 
    ufw default deny incoming && ufw default allow outgoing
ufw allow 22/tcp
# 80/443 só dos ranges da Cloudflare:
for ip in $(curl -s https://www.cloudflare.com/ips-v4) $(curl -s https://www.cloudflare.com/ips-v6); do
  ufw allow from $ip to any port 80 proto tcp
  ufw allow from $ip to any port 443 proto tcp
done
ufw --force enable
🧪 Teste: ufw status → "Status: active" e regras de 80/443 com origens da Cloudflare. fail2ban-client status → serviço ativo.

3.4 SSH só por chave (⛔ cuidado pra não se trancar)

  1. Edite /etc/ssh/sshd_config e garanta: 
    PermitRootLogin no
PasswordAuthentication no
  2. Antes de reiniciar: abra outro PowerShell e teste ssh deploy@5.6.7.8.
  3. Só se entrou como deploy: systemctl restart ssh.
🧪 Teste: num terminal novo, ssh deploy@IP entra; ssh root@IP é recusado (root desligado).

3.5 Instalar Docker

# como deploy
curl -fsSL https://get.docker.com | sudo sh
sudo usermod -aG docker deploy
exit   # saia e entre de novo p/ valer o grupo
🧪 Teste: após reconectar como deploy → docker run hello-world → imprime "Hello from Docker!".

4.3 docker-compose.yml

nano docker-compose.yml
services:
  web:
    build: { context: ./backend }
    env_file: .env
    expose: ["5000"]
    restart: unless-stopped

  nginx:
    image: nginx:alpine
    ports: ["80:80", "443:443"]
    volumes:
      - ./deploy/nginx.conf:/etc/nginx/conf.d/default.conf:ro
      - ./deploy/cloudflare-ips.conf:/etc/nginx/conf.d/cloudflare-ips.conf:ro
      - /etc/ssl/cloudflare:/etc/ssl/cloudflare:ro
    depends_on: [web]
    restart: unless-stopped

4.4 Nginx + placeholder de IPs

mkdir -p deploy
echo "# preenchido na Fase 5.3" > deploy/cloudflare-ips.conf
nano deploy/nginx.conf
server {
  listen 80;
  server_name api.grindlabpoker.com.br;
  return 301 https://$host$request_uri;
}
server {
  listen 443 ssl;
  http2 on;
  server_name api.grindlabpoker.com.br;

  ssl_certificate     /etc/ssl/cloudflare/origin.pem;
  ssl_certificate_key /etc/ssl/cloudflare/origin.key;
  ssl_protocols TLSv1.2 TLSv1.3;

  include /etc/nginx/conf.d/cloudflare-ips.conf;
  real_ip_header CF-Connecting-IP;
  client_max_body_size 6m;

  location / {
    proxy_pass http://web:5000;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_read_timeout 180s;
  }
}

Na zona grindlabpoker.com.br.

9Solver GTO — a 2ª máquina ~30 min · quando for ligar o GTO

Máquina separada, só pra solves. Nunca fica pública: o backend a acessa por rede privada do Hetzner. (Pode deixar pra depois — o app funciona sem GTO; GTO_SOLVER_URL vazio.)

9.1 Rede privada (liga as duas máquinas)

  1. Hetzner → Networks → Create Network: nome grindlab-net, range 10.0.0.0/16.
  2. Anexe os dois servidores a essa rede (o backend grindlab-api e o solver, abaixo). Cada um ganha um IP interno 10.0.0.x.

9.2 Criar a máquina do solver

  1. Hetzner → Add Server, mesma Location do backend (a rede privada exige mesma região).
  2. Type: CX43 (8 vCPU / 16 GB) p/ 1 solve por vez; CX52 (16 vCPU / 32 GB) p/ 2–3 simultâneos.
  3. Image: Ubuntu 24.04 · SSH key: a sua · Network: marque grindlab-net · Name: grindlab-solver.
  4. Anote o IP interno (ex.: 10.0.0.3).

9.3 Travar o solver (nunca público)

  1. Endureça igual à Fase 3.3/3.4 (ufw, fail2ban, SSH só-chave, Docker).
  2. Firewall: bloqueie TODO ingresso público. Libere só: SSH (22) do seu IP; e a porta do solver (ex.: 8000) só do IP interno do backend: 
    sudo ufw default deny incoming && sudo ufw default allow outgoing
sudo ufw allow from SEU_IP to any port 22 proto tcp
sudo ufw allow from 10.0.0.2 to any port 8000 proto tcp   # 10.0.0.2 = IP interno do backend
sudo ufw --force enable
AGPL: o solver (postflop-solver) é AGPL — roda server-side only, nunca é distribuído nem exposto. Esta máquina fechada cumpre isso. (No backend, o /gto/strategy já é read-only por padrão — solve ao vivo só admin.)

9.4 Compilar e subir o solver

O solver é um binário Rust (compila na própria máquina). Resumo (o detalhamento fino fica como add-on):

cd ~ && git clone https://github.com/wrodrigobr/Leaklab.git app && cd app
# instalar Rust + compilar o solver_cli (postflop-solver)
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh -s -- -y
source $HOME/.cargo/env
cd backend/gto_bot/solver_cli && cargo build --release
# subir o serviço HTTP do solver (solver_api) — escuta em 0.0.0.0:8000 (já travado pelo firewall)
Concorrência: rode o worker do solver com 1 solve por vez e nice (prioridade baixa) — protege CPU/RAM. Para uma campanha de precompute (dias a 100%), use uma máquina temporária maior e desligue depois.

9.5 Ligar no backend

Na máquina do backend, edite o .env:

GTO_SOLVER_URL=http://10.0.0.3:8000    # IP interno do solver
docker compose up -d   # aplica
🧪 Teste: no backend, curl http://10.0.0.3:8000/ responde (rede privada ok). De fora da VPC, o mesmo IP não responde (travado). No app, um spot postflop sem cobertura passa a ser resolvido em background.

⚠ Troubleshooting

SintomaCausa / solução
Domínio não abreNameservers propagando (até 24h) ou zona não Active. Confira nslookup -type=NS ….
Cloudflare 521/522Origem não respondeu: nginx parado (docker compose ps), firewall não libera a CF, ou cert ausente.
525 SSL handshakeFalta Origin Cert no Nginx ou SSL não está Full (strict).
CORS no loginALLOWED_ORIGINS errado → corrija + docker compose up -d.
Backend não sobe (DB)DATABASE_URL sem ?sslmode=require ou Neon pausado.
Webhook Stripe falhaURL errada no setup, ou WAF/Bot desafiando → exceção (Skip) p/ /subscription/webhook.
Trancado fora do SSHUse o Console do servidor no painel Hetzner (VNC) e conserte sshd_config/ufw.
Logs: docker compose logs -f web e docker compose logs -f nginx revelam quase tudo.

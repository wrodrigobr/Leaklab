Guia de Deploy — GrindLab

Cloudflare (frontend + proxy) · Neon (Postgres gerenciado) · GCP (backend, usando créditos — Hetzner depois). Domínios grindlabpoker.com.br (canônico) e grindlabpoker.com (→ 301).

Visão geral

3 serviços, cada um com seu papel. O banco no Neon já resolve o backup automático.

Usuário │ ▼ ┌─────────────── Cloudflare ───────────────┐ │ Pages (frontend estático) + Proxy/CDN │ └───────┬───────────────────────┬──────────┘ │ grindlabpoker.com.br │ api.grindlabpoker.com.br ▼ ▼ [ SPA React ] ┌──────────────────┐ ┌───────────────┐ (na borda CF) │ GCP VM (créditos)│ SSL │ Neon │ │ Nginx → Gunicorn │──────▶ │ PostgreSQL │ │ (Flask, Docker) │ │ (gerenciado, │ └────────┬─────────┘ │ backup auto) │ │ IP interno (VPC) └───────────────┘ ▼ Solver (AGPL) = VM à parte no MESMO projeto GCP (rede privada, nunca público).
Por que assim: o frontend é estático → vai pro Pages (grátis, CDN global, TLS, zero-ops). O backend precisa de servidor → Hetzner. O banco no Neon tem backup automático, eliminando o maior risco operacional.
Ordem importa: faça na sequência das seções (1→8). A propagação de DNS é o gargalo — comece por ela.

0Contas que você vai precisar

  • Cloudflare — dash.cloudflare.com (plano Free).
  • Neon — neon.tech (Postgres gerenciado, free tier).
  • Google Cloud (GCP) — console.cloud.google.com — usaremos os créditos que você já tem para o backend agora (o solver já vive lá). Hetzner fica como migração futura.
  • Stripe — dashboard.stripe.com (modo test primeiro).
  • GitHub — o repositório do projeto (o Pages faz deploy a partir dele).
  • Anthropic — a ANTHROPIC_API_KEY (Claude Haiku, usado no app).
Economia agora: rodamos o backend numa VM do GCP com seus créditos (custo ~zero no começo). Quando os créditos acabarem, migrar pro Hetzner é só refazer a Fase 3 noutra máquina — o resto (Cloudflare, Neon, Pages, Stripe) não muda.
Chave SSH: se ainda não tem, gere uma: ssh-keygen -t ed25519 -C "deploy" (gera ~/.ssh/id_ed25519.pub).

1Cloudflare — domínios ~10 min + propagação

Adicione as 2 zonas e aponte os nameservers. O .com já propagou ✔ — falta o .com.br.

1.1 Adicionar as zonas

  1. Cloudflare → Add a site → grindlabpoker.com.br → plano Free.
  2. Repita para grindlabpoker.com (cada domínio é uma zona separada).
  3. Cada zona te dá 2 nameservers (ex.: xxx.ns.cloudflare.com).

1.2 Trocar os nameservers (no registrador)

  • .com.br → registro.br → painel do domínio → Alterar servidores DNS → cole os 2 da Cloudflare.
  • .com → já está Active ✔ (feito).

1.3 Definir o canônico + redirect

O canônico é o .com.br (público BR). Na zona do .com: Rules → Redirect Rules → Create:

# Quando: Hostname é grindlabpoker.com OU www.grindlabpoker.com
When incoming requests match:  Hostname  equals  grindlabpoker.com
Then:  Redirect → Type 301 → Dynamic
       Expression:  concat("https://grindlabpoker.com.br", http.request.uri.path)
       Preserve query string: ON
Enquanto o .com.br não propaga: você pode adiantar tudo testando pelo .com (que já está ativo) e só ligar o redirect/canônico quando o .com.br virar Active.

2Neon — Postgres gerenciado ~5 min

Cria o banco e te dá uma DATABASE_URL. Backup automático incluso.

  1. Em neon.tech → Create project.
  2. Region: a mesma do backend e dos usuários — como a VM do GCP fica em São Paulo, escolha AWS São Paulo (sa-east-1) no Neon (menor latência banco↔backend).
  3. Anote a Connection string (use a Pooled connection). Formato:
postgresql://USUARIO:SENHA@ep-xxxx-pooler.sa-east-1.aws.neon.tech/grindlab?sslmode=require
É só isso. O app detecta DATABASE_URL e usa Postgres automaticamente; as migrações rodam sozinhas no startup (cria todas as tabelas). Sem DATABASE_URL ele cairia em SQLite — então não esqueça de setar.
Backup (a sua preocupação): o Neon mantém histórico/branching e restauração point-in-time (no free tier o retentivo é curto; nos pagos, vários dias). Guarde também a DATABASE_URL e o .env num cofre (não no git).

3GCP — VM do backend (créditos) ~20 min

Usamos uma VM do Compute Engine com seus créditos. O solver já está no GCP → backend e solver podem conversar por IP interno (VPC), sem expor o solver.

Migrar pro Hetzner depois é só refazer esta fase numa VM Hetzner (Ubuntu + Docker são idênticos); 3.2–3.4 valem igual. As diferenças são só a provisão (3.1) e o firewall do GCP (3.5).

3.1 Criar a VM (Compute Engine)

  1. Compute Engine → Create Instance (no mesmo projeto do solver).
  2. Region: southamerica-east1 (São Paulo) — perto dos usuários e do Neon SP. Machine: e2-medium (2 vCPU / 4 GB) ou e2-small (o banco fica no Neon, então RAM é folgada).
  3. Boot disk: Ubuntu 24.04 LTS, 20–30 GB.
  4. Firewall: NÃO marque "Allow HTTP/HTTPS" genérico (vamos restringir só à Cloudflare em 3.5). Permita só o necessário.
  5. SSH: em Security → Manage SSH keys cole sua chave pública (ou use gcloud compute ssh).

3.2 IP externo estático

Reserve um IP estático (senão muda ao parar/iniciar a VM e quebra o DNS):

# VPC network → IP addresses → Reserve external static IP → anexe à VM
# anote esse IP — é a "origem" no Cloudflare (Fase 5)

3.3 Endurecer o servidor

# conecta (gcloud ou ssh direto)
gcloud compute ssh SUA_VM --zone southamerica-east1-a    # ou: ssh USUARIO@IP

# usuário sudo dedicado (se entrou como outro) + patches + ferramentas
sudo apt update && sudo apt -y upgrade
sudo apt -y install ufw fail2ban unattended-upgrades
sudo dpkg-reconfigure -plow unattended-upgrades   # confirme "Yes"

# firewall do SO (2ª camada; a 1ª é o firewall do GCP em 3.5)
sudo ufw default deny incoming && sudo ufw default allow outgoing
sudo ufw allow 22/tcp
sudo ufw allow 443/tcp && sudo ufw allow 80/tcp   # restringidos à CF na Fase 5.3
sudo ufw --force enable

3.4 SSH só por chave + Docker

# /etc/ssh/sshd_config: PasswordAuthentication no  →  sudo systemctl restart ssh
# (teste o login num novo terminal ANTES de fechar a sessão atual)

# Docker
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER   # saia e entre de novo

3.5 Firewall do GCP (1ª camada) — só Cloudflare

No GCP o firewall é na VPC (antes do SO). Crie a regra de ingresso permitindo 80/443 só dos ranges da Cloudflare e 22 só do seu IP:

# ranges da Cloudflare numa variável
CF=$(curl -s https://www.cloudflare.com/ips-v4 | paste -sd, -),$(curl -s https://www.cloudflare.com/ips-v6 | paste -sd, -)

gcloud compute firewall-rules create allow-cloudflare-web \
  --direction=INGRESS --action=ALLOW --rules=tcp:80,tcp:443 \
  --source-ranges="$CF" --target-tags=grindlab-web

gcloud compute firewall-rules create allow-ssh-me \
  --direction=INGRESS --action=ALLOW --rules=tcp:22 \
  --source-ranges="SEU_IP/32" --target-tags=grindlab-web

# marque a VM com a tag de rede 'grindlab-web' (Console → VM → Network tags)
# e garanta que NÃO há regra default liberando 0.0.0.0/0 em 80/443
Solver no mesmo projeto: não exponha o solver publicamente. No .env (Fase 4) aponte GTO_SOLVER_URL para o IP interno da VM do solver (ex.: http://10.x.x.x:8000) — tráfego só dentro da VPC.

4Backend — subir no VPS ~30 min

4.1 Clonar o repositório

cd /home/deploy
git clone https://github.com/wrodrigobr/Leaklab.git app
cd app

4.2 Criar o .env (segredos — fora do git!)

cat > .env <<'EOF'
ENVIRONMENT=production
LEAKLAB_SECRET=<gere 40+ chars aleatórios>
DATABASE_URL=postgresql://...neon...?sslmode=require
ANTHROPIC_API_KEY=sk-ant-...
ALLOWED_ORIGINS=https://grindlabpoker.com.br,https://www.grindlabpoker.com.br,https://grindlabpoker.com
STRIPE_SECRET_KEY=sk_live_...
STRIPE_WEBHOOK_SECRET=whsec_...
STRIPE_PRICE_PRO_MONTHLY=price_...
STRIPE_PRICE_PRO_ANNUAL=price_...
GTO_SOLVER_URL=
PORT=5000
EOF
chmod 600 .env
Gerar o LEAKLAB_SECRET: openssl rand -base64 48. (As vars do Stripe você preenche na Fase 7; pode deixar vazias por ora — o app sobe mesmo assim.)

4.3 docker-compose (backend + nginx)

Crie /home/deploy/app/docker-compose.yml:

services:
  web:
    build: { context: ./backend }
    env_file: .env
    expose: ["5000"]
    restart: unless-stopped

  nginx:
    image: nginx:alpine
    ports: ["80:80", "443:443"]
    volumes:
      - ./deploy/nginx.conf:/etc/nginx/conf.d/default.conf:ro
      - /etc/ssl/cloudflare:/etc/ssl/cloudflare:ro
    depends_on: [web]
    restart: unless-stopped

4.4 Nginx (reverse-proxy do api)

Crie /home/deploy/app/deploy/nginx.conf (o certificado vem na Fase 5):

server {
  listen 80;
  server_name api.grindlabpoker.com.br;
  return 301 https://$host$request_uri;
}
server {
  listen 443 ssl;
  http2 on;
  server_name api.grindlabpoker.com.br;

  ssl_certificate     /etc/ssl/cloudflare/origin.pem;
  ssl_certificate_key /etc/ssl/cloudflare/origin.key;
  ssl_protocols TLSv1.2 TLSv1.3;

  # IP real do cliente (Cloudflare) — ranges em cloudflare-ips.conf (Fase 5)
  include /etc/nginx/conf.d/cloudflare-ips.conf;
  real_ip_header CF-Connecting-IP;

  client_max_body_size 6m;   # uploads de hand history

  location / {
    proxy_pass http://web:5000;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_read_timeout 180s;   # análises/relatórios podem demorar
  }
}

4.5 Subir

docker compose up -d --build
docker compose logs -f web   # confira: migrações aplicadas, "Listening"
As migrações do banco rodam sozinhas no startup (cria as tabelas no Neon). Teste rápido de dentro do VPS: curl -k https://localhost/health deve responder OK.

5Cloudflare na frente do api ~15 min

TLS correto + esconder a origem + IP real. Tudo na zona grindlabpoker.com.br.

5.1 DNS

DNS → Records → Add record: A · nome api · conteúdo IP do VPS · Proxied 🟠.

5.2 SSL: Full (Strict) + Origin Certificate

  1. SSL/TLS → Overview → Full (Strict). Nunca "Flexible".
  2. SSL/TLS → Origin Server → Create Certificate → gera 2 textos (cert + key). No VPS:
sudo mkdir -p /etc/ssl/cloudflare
sudo nano /etc/ssl/cloudflare/origin.pem   # cole o certificado
sudo nano /etc/ssl/cloudflare/origin.key   # cole a chave privada
sudo chmod 600 /etc/ssl/cloudflare/origin.key

SSL/TLS → Edge Certificates: ligue Always Use HTTPS, Min TLS 1.2, e HSTS (depois de confirmar tudo em HTTPS).

5.3 IP real (Nginx) — a origem só-CF você já fez no GCP (3.5)

Gere o arquivo de ranges da Cloudflare p/ o Nginx restaurar o IP real do cliente:

# ranges oficiais → include do nginx (set_real_ip_from)
( for ip in $(curl -s https://www.cloudflare.com/ips-v4) $(curl -s https://www.cloudflare.com/ips-v6); do
    echo "set_real_ip_from $ip;"; done ) | sudo tee /home/deploy/app/deploy/cloudflare-ips.conf

# monte esse arquivo no nginx (adicione o volume no compose):
#   - ./deploy/cloudflare-ips.conf:/etc/nginx/conf.d/cloudflare-ips.conf:ro
docker compose restart nginx
Bloqueio só-Cloudflare: no GCP isso é a regra de firewall da VPC (Fase 3.5, allow-cloudflare-web) — é a 1ª camada. O ufw da Fase 3.3 é a 2ª camada (defesa em profundidade). No Hetzner, seria só o ufw restrito aos ranges CF.
IP real já tratado no app: o backend tem ProxyFix e o nginx restaura CF-Connecting-IP → o rate-limiter anti-abuso enxerga o IP do usuário (não o da Cloudflare).

5.4 Cache

Rules → Cache Rules: para api.grindlabpoker.com.br/* → Bypass cache (API é dinâmica; nunca cachear, principalmente /subscription/webhook).

6Frontend — Cloudflare Pages ~15 min

  1. Cloudflare → Workers & Pages → Create → Pages → Connect to Git → repositório do projeto.
  2. Build: Framework Vite · Build command cd frontend && npm install && npm run build · Output frontend/dist.
  3. Environment variables (Production): 
    VITE_API_URL=https://api.grindlabpoker.com.br
VITE_STRIPE_PUBLISHABLE_KEY=pk_live_...
  4. Custom domains: adicione grindlabpoker.com.br e www ao projeto Pages (cria os registros sozinho).
  5. Cada git push → build + deploy automático na borda.
Frontend e API em subdomínios diferentes → o CORS já está coberto pela ALLOWED_ORIGINS que você pôs no .env (Fase 4.2).

7Stripe — recorrência ~15 min

Um script já cria Product, Prices recorrentes, webhook e Billing Portal.

# no VPS, dentro do container web (ou local com a venv):
docker compose exec web sh -lc '
  export STRIPE_SECRET_KEY=sk_live_xxx
  python scripts/stripe_setup.py \
    --webhook-url https://api.grindlabpoker.com.br/subscription/webhook --apply '

Ele imprime os IDs. Cole no .env e reinicie:

STRIPE_PRICE_PRO_MONTHLY=price_...
STRIPE_PRICE_PRO_ANNUAL=price_...
STRIPE_WEBHOOK_SECRET=whsec_...
# depois
docker compose up -d
  • No Stripe Dashboard → Customer portal: confirme que está ativado (o script já cria a config).
  • Cloudflare: garanta o Bypass cache em /subscription/webhook e, se tiver Bot/WAF agressivo, uma exceção (Skip) pra essa rota (senão o Stripe é desafiado e os eventos falham).
Teste em sk_test_ primeiro. Rode o setup com a chave de teste, valide o fluxo de assinatura, e só então repita com sk_live_.

8Crons & manutenção ~15 min

8.1 Jobs diários (billing + coach)

No VPS, crontab -e do usuário deploy:

# 04:10 — expira assinaturas/cortesias vencidas
10 4 * * * cd /home/deploy/app && docker compose exec -T web python scripts/expire_subscriptions.py
15 4 * * * cd /home/deploy/app && docker compose exec -T web python scripts/expire_coach_trials.py

8.2 Atualizações (CF IPs + sistema)

# 1x/mês — atualiza os ranges da Cloudflare (mudam raramente)
0 5 1 * * cd /home/deploy/app && bash deploy/refresh_cloudflare_ips.sh

8.3 Backup

  • Banco: o Neon faz backup/PITR automático — sua maior preocupação está coberta.
  • VPS: o código está no git; só o .env e os certs precisam de cópia segura (guarde num cofre de senhas).
  • Extra (opcional): snapshot semanal do disco da VM no GCP (Compute Engine → Snapshots; agende um snapshot schedule) para restaurar a máquina inteira rápido. No Hetzner, seria o Snapshot do painel.

8.4 Deploy de novas versões

cd /home/deploy/app && git pull && docker compose up -d --build
